# 主动变更驱动开发用户说明（Active Change）

> 本文件是给用户看的说明，不是规范文件，不是流程文件，也不是门禁文件。
> 它只用来解释 OpenSpec 在这个仓库里的使用方式。

本文件面向使用者，说明 OpenSpec 在项目中的工作方式。目标是把 OpenSpec 作为变更工作台，而不是事后归档仓库。

## 1. 新建与变更规则
- 先规划后实施。
- 同一意图、同一主路径的收敛修正，优先更新当前 change。
- 需求边界变化明显时，必须新建 change。

## 2. 认知回写机制
- 实施过程中如果发现理解有偏差，优先回写到当前 change 的 proposal / design / specs / tasks。
- 不要只改代码而不回写文档。

## 3. Archive 时机定义
- `archive` 只用于 change 收口完成并准备并入正式规范库。
- 执行前先用 `openspec list --json` 或 `openspec status --change "<name>" --json` 确认目标 change。
- 只有当以下条件全部满足时，才允许执行 archive：
  1. 该 change 对应的实现范围已经完成。
  2. 代码与 proposal / design / specs / tasks 已重新对齐。
  3. 所有承诺的验证命令已执行并通过。
  4. 实现中新发现的重要认知已经回写到 change 文档。
  5. 归档后主 spec 不会退化。
- 对未完成 artifacts 或未完成 `tasks.md`，只发出警告，不手工补做归档前收口。
- 默认直接执行 `openspec archive "<name>" --yes`。
- 只有明确属于 docs/tooling 且不影响主 spec 的 change，才允许使用 `--skip-specs`。
- 不再手工 merge delta specs，也不再手工移动 `openspec/changes/...`。

## 4. 实施阶段定位
- OpenSpec 负责规划、约束、追踪与归档安全。
- 代码编写、测试执行、交付规范仍由项目自己的实施流程负责。
